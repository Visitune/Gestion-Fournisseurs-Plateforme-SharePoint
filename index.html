Les 5 problèmes concrets

Problème 1 — Le document fantôme

Un fournisseur envoie une nouvelle fiche technique. Elle atterrit dans les mails, quelqu'un la copie sur le serveur sous un nom différent. L'ancienne fiche technique reste là aussi. On ne sait plus laquelle est valide. Résultat : risque de travailler avec un document périmé.

Problème 2 — La surprise d'audit

L'auditeur IFS/BRC demande le certificat FSSC 22000 du fournisseur X. Il est expiré depuis 3 mois. Personne ne le savait. Non-conformité majeure.

Problème 3 — L'Excel ingérable

Un fichier Excel avec 60+ colonnes, 200 lignes, des formules qui se cassent, plusieurs personnes qui travaillent dessus en même temps. Erreurs garanties, perte de temps assurée.

Problème 4 — La relance manuelle

Chaque mois, quelqu'un passe 2-3 heures à vérifier manuellement les dates et envoyer des emails aux fournisseurs pour leur demander de mettre à jour leurs documents. Travail chronophage et source d'oublis.

Problème 5 — Pas de vision globale

Impossible de répondre en 2 minutes à la question : "Combien de fournisseurs sont actuellement conformes ?" Il faut ouvrir l'Excel, filtrer, compter… Pas de pilotage possible.

🔄 Ce qui se passe automatiquement chaque jour

1

08h00 — Le système vérifie toutes les dates d'expiration

Chaque matin, Power Automate compare les dates d'expiration de tous les documents avec la date du jour. Si un document arrive à échéance dans 90, 30 ou 7 jours, il met à jour son statut automatiquement.

2

08h30 — Les emails d'alerte sont envoyés

Pour les documents qui tombent pile à J-90, J-30 ou J-7 ce jour-là, un email automatique est envoyé aux personnes concernées. Un seul email par document, pas de spam quotidien.

3

Sur demande — Relance fournisseur en 1 clic

Un responsable Qualité ou Achats clique sur "Relancer ce fournisseur" dans la fiche. Un email professionnel est envoyé automatiquement au contact du fournisseur, avec la liste exacte des documents manquants ou expirés.

4

À la réception — Nouveau document = ancien archivé

Quand un fournisseur envoie un nouveau certificat, l'ancien passe automatiquement en "Obsolète". Il reste visible dans l'historique mais ne génère plus d'alerte. La version courante est toujours évidente.

📂 Classification automatique des fichiers

Dans SharePoint, les documents ne sont pas jetés dans un dossier "vrac". Ils sont classés tout seuls dès qu'on les dépose.

Par Fournisseur

Chaque fournisseur a son propre dossier parent.

Par Matière

À l'intérieur, les fiches sont classées par produit.

Par Année / Version

L'historique est conservé proprement.

📎 3 niveaux de rattachement

Un document peut être lié à différents niveaux selon sa nature. C'est important car ça détermine qui est concerné par les alertes.

Niveau Fournisseur

Le document concerne le fournisseur lui-même, indépendamment des matières qu'il fournit.

Exemples :
Certificat IFS / BRC / FSSC 22000
Questionnaire fournisseur
Évaluation annuelle fournisseur
Niveau Matière

Le document concerne la matière en général, peu importe d'où elle vient.

Exemples :
Fiche technique générique
Cahier des charges interne
Spécification produit
Fournisseur + Matière

Le document est spécifique à cette source pour cette matière.

Exemples :
FT spécifique au fournisseur
Bulletin d'analyse labo
Déclaration alimentarité

🆕 Ce qui se passe quand on reçoit un nouveau document

1

Le fournisseur envoie sa nouvelle fiche technique

Via email, ou via le portail de dépôt. Le fichier est reçu dans SharePoint.

2

Le document passe en "En attente de validation"

La personne Qualité reçoit une notification. Elle consulte le document dans SharePoint.

3

La personne Qualité valide le document

Elle clique sur "Valider" — le document passe à "Valide". La date de validation et son nom sont enregistrés automatiquement.

4

L'ancienne version passe automatiquement en "Obsolète"

Le système détecte qu'il existait déjà une fiche technique pour ce fournisseur + cette matière. L'ancienne passe en Obsolète, la nouvelle devient "version courante". L'historique est conservé.

🔔 Les alertes automatiques

Plus jamais de certificat expiré sans que personne ne le sache. Le système surveille tout et envoie les bons emails aux bonnes personnes.

📅 Quand envoie-t-on une alerte ?

J-90

3 mois avant expiration

Email envoyé au service Qualité et Achats.
Objectif : anticiper, contacter le fournisseur, demander le renouvellement sans urgence.

J-30

1 mois avant expiration

Email envoyé au service Qualité, Achats, et au contact qualité du fournisseur.
Objectif : le fournisseur est maintenant informé directement. Il doit agir.

J-7

⚠️ URGENT — 7 jours avant expiration

Email envoyé à tous les destinataires, avec le sujet préfixé "⚠ URGENT".
Objectif : dernière chance avant non-conformité.

0 jours

Le document est EXPIRÉ

Alerte maximale. Le système marque le fournisseur comme "Non-Conforme" et bloque potentiellement les commandes selon vos règles.

💡

Important : les alertes sont envoyées une seule fois, le jour exact (J-90, J-30, etc.). Pas d'email répété chaque jour pour éviter de noyer les boîtes mail. Le tableau de bord SharePoint, lui, montre en permanence l'état réel.

⚙️ Les 5 flux automatiques (Power Automate)

Power Automate est l'outil qui fait tourner toutes les règles automatiques. Voici les 5 flux configurés :

🔄 Flux 1 — Calcul statuts

Tourne tous les jours à 08h00. Met à jour le statut de chaque document actif en comparant sa date d'expiration avec aujourd'hui.

🔔 Flux 2 — Alertes expiration

Tourne tous les jours à 08h30 (après le flux 1). Envoie les emails pour les documents qui arrivent à J-90, J-30 ou J-7 ce jour précis.

✉️ Flux 3 — Relance fournisseur

Déclenché manuellement par un clic sur le bouton "Relancer". Génère et envoie l'email de relance avec la liste des documents à renouveler.

📄 Flux 4 — Nouveau document reçu

Déclenché automatiquement quand un nouveau document est ajouté. Archive l'ancien, incrémente le numéro de version du nouveau.

📥 Flux 5 — Dépôt fournisseur

Déclenché quand un fournisseur dépose un fichier via le portail. Notifie la Qualité et crée l'entrée en "En attente de validation".

Phase 0 — Fondations

Mise en place de la structure SharePoint. Durée totale : 1h à 2h pour un nouveau client.

Avant de commencer, vérifiez que vous avez :
  • Un compte administrateur sur le tenant Microsoft 365 du client (ou les droits pour créer des sites SharePoint)
  • Le site SharePoint cible déjà créé — le script ne crée pas le site, il y installe la structure. Exemple d'URL : https://client.sharepoint.com/sites/GestionFournisseurs
  • Votre ordinateur personnel (pas un serveur cloud) — le script doit tourner en local
  • Connexion internet active

Étape A — Préparer votre poste (à faire une seule fois)

A1

Installer PowerShell 7

PowerShell est l'outil en ligne de commande fourni par Microsoft qui va exécuter nos scripts. La version 7 (et non la version Windows intégrée) est obligatoire.

Comment l'installer :

  1. Appuyez sur Windows + R, tapez winget install Microsoft.PowerShell dans une fenêtre PowerShell classique, appuyez sur Entrée
  2. Attendez la fin de l'installation (2-3 minutes)
  3. Fermez et rouvrez PowerShell — vous devriez voir PS 7.x dans le titre de la fenêtre
winget install Microsoft.PowerShell

✅ Pour vérifier : tapez $PSVersionTable.PSVersion — la version doit afficher 7.4.6 ou supérieur.

A2

Installer le module PnP PowerShell

PnP PowerShell est une bibliothèque de commandes qui permet de piloter SharePoint depuis le terminal. C'est elle qui fait le travail concret de création des listes.

Dans PowerShell 7, copiez-collez cette commande :

Install-Module PnP.PowerShell -Scope CurrentUser

Si une fenêtre vous demande de confirmer l'installation depuis la galerie Microsoft, tapez O puis Entrée. L'installation prend 1-2 minutes.

Étape B — Configurer l'accès au tenant client (une fois par client)

B1

Récupérer le nom du tenant

Le "tenant" est l'identifiant de l'organisation Microsoft 365 du client. Vous le trouvez dans l'URL de son SharePoint.

Exemple : si l'URL SharePoint du client est
https://acmefood.sharepoint.com/sites/...
alors le tenant est acmefood.onmicrosoft.com

📝 Notez ce nom, vous en aurez besoin à l'étape suivante.

B2

Enregistrer l'application d'accès (Entra ID) ⚠️ obligatoire depuis sept. 2024

Avant de pouvoir piloter SharePoint depuis un script, Microsoft impose d'enregistrer une "application" dans leur annuaire (Entra ID, anciennement Azure AD). C'est une sorte de carte d'identité pour le script. Cette étape se fait une seule fois par client.

Dans PowerShell 7, remplacez le tenant et exécutez :

Register-PnPEntraIDAppForInteractiveLogin `
  -ApplicationName "PnP-FRS" `
  -SharePointDelegatePermissions "AllSites.FullControl" `
  -Tenant "acmefood.onmicrosoft.com" `
  -Interactive
Ce qui va se passer :
  1. Une fenêtre de connexion Microsoft s'ouvre — connectez-vous avec un compte admin du client
  2. Microsoft vous demande de confirmer les autorisations — cliquez Accepter
  3. Dans le terminal, vous voyez s'afficher un AppID (un code de 36 caractères) :
    AppId: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx

⚠️ Copiez et sauvegardez cet AppID dans un fichier texte. Vous en aurez besoin à l'étape suivante.

Étape C — Déployer SharePoint

C1

Ouvrir le dossier du projet dans PowerShell

Le script doit être exécuté depuis le dossier où il se trouve. Dans PowerShell 7 :

cd "C:\chemin\vers\PROJET SHAREPOINT FRS\scripts"

Astuce : dans l'Explorateur Windows, naviguez dans le dossier scripts/, puis cliquez sur la barre d'adresse, tapez pwsh et Entrée — PowerShell 7 s'ouvre directement dans ce dossier.

C2

Lancer le script de déploiement

Remplacez l'URL et l'AppID par les vraies valeurs du client, puis exécutez :

./Deploy-FRS.ps1 `
  -SiteUrl "https://acmefood.sharepoint.com/sites/GestionFournisseurs" `
  -ClientId "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"

Le fichier JSON client définit exactement quels types de documents créer et si le client suit ses matières premières individuellement (niveau Fournisseur + Matière) ou seulement ses fournisseurs (niveau Fournisseur seul). Étape D — Configurer les flux Power Automate

D1

Récupérer les IDs des listes SharePoint

Chaque liste SharePoint a un identifiant unique (GUID) que vous devrez coller dans les flux. Pour le trouver :

  1. Ouvrez la liste dans SharePoint (ex: Documents)
  2. Cliquez sur l'engrenage ⚙️ → Paramètres de la liste
  3. Dans la barre d'adresse du navigateur, cherchez le paramètre List=
  4. Copiez la valeur entre les accolades : {xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx}

Répétez pour les listes : Documents, Fournisseurs, Types_Documents, Matieres_Premieres.

D2

Créer le formulaire Microsoft Forms

Le formulaire sert de portail de dépôt pour les fournisseurs (Flux 5 uniquement). À créer une seule fois sur forms.microsoft.com avec votre compte Microsoft 365.

Créez un nouveau formulaire avec ces 5 questions dans cet ordre :
  1. Code fournisseur — Texte (obligatoire)
  2. Type de document — Liste déroulante (obligatoire) — options = vos types de documents
  3. Matière première concernée — Texte (optionnel)
  4. Date d'émission du document — Date (obligatoire)
  5. Fichier à déposer — Téléchargement de fichier (obligatoire)

Une fois créé, notez l'ID du formulaire visible dans l'URL : forms.microsoft.com/r/VOTRE_FORM_ID

D3

Importer les 5 flux dans Power Automate

Rendez-vous sur make.powerautomate.com avec le compte du client. Pour chacun des 5 fichiers JSON :

  1. Cliquez Mes flux → Nouveau flux → choisissez le type correspondant (planifié pour Flux 1, automatisé pour les autres)
  2. Dans l'éditeur, cliquez sur le bouton … (trois points) en haut à droite → Modifier en JSON (code)
  3. Sélectionnez tout le contenu Ctrl+A, supprimez-le, puis collez le contenu du fichier JSON correspondant
  4. Cliquez Enregistrer
1 flow_calcul_statuts.json — Calcul des statuts · tourne chaque matin à 8h
2 flow_alertes_expiration.json — Emails d'alerte J-90 / J-30 / J-7
3 flow_relance_fournisseur.json — Relance manuelle depuis une fiche fournisseur
4 flow_nouveau_document.json — Archivage + versioning automatique
5 flow_portail_fournisseur.json — Portail de dépôt via Microsoft Forms
D4

Configurer les paramètres dans chaque flux

🚀 Script de déploiement SharePoint

Deploy-FRS.ps1
Script PnP PowerShell v3 — crée les 6 listes + bibliothèque, colonnes, vues filtrées et 8 types de documents par défaut en une seule exécution.
Phase 0 · Requis PS 7.4.6+

Phase 0 — Fondations

Prérequis M365 !
1 Créer l'espace client sur Microsoft 365

Avant de lancer quoi que ce soit, il faut simplement créer un nouveau site SharePoint vierge pour le client (modèle "Site d'équipe" régulier).

🔗 L'URL de ce site sera essentielle pour la suite. Exemple : https://votre-tenant.sharepoint.com/sites/GestionFournisseurs
2 Générer une "clé secrète" (Application Entra ID)

Pour que notre outil puisse créer l'architecture tout seul, il lui faut une permission spéciale de Microsoft. C'est ce qu'on appelle "Enregistrer une application Entra ID". Ne vous inquiétez pas, il y a un petit bout de code automatique pour ça (à ne lancer qu'une seule fois par compte Microsoft).

En pratique : Cette étape vous donnera un grand code chiffré appelé "ID de Client" (Client ID). Notez-le bien, c'est la clé magique pour l'étape suivante !

3 La magie opère : Le script "Deploy-FRS"

C'est le moment satisfaisant. Vous ouvrez PowerShell, vous lancez le fichier Deploy-FRS.ps1 en lui donnant l'URL de l'étape 1 et la clé secrète de l'étape 2. Le script prend le contrôle :

  • Création de 6 listes SharePoint (Fournisseurs, Documents, Matières...)
  • Création d'une bibliothèque de stockage pour les futurs fichiers PDF/Excel
  • Configuration de toutes les colonnes expertes (Statuts avec puces de couleur, formules de retard...)
  • Aucun clic manuel requis. Durée : environ 3 minutes.
4 Brancher le "cerveau" automatique (Flux Power Automate)

L'infrastructure est prête, mais elle est encore "endormie". Pour qu'elle s'auto-gère, on va importer 4 petits fichiers dans l'outil Power Automate (qui est déjà compris dans la licence client).

Flux 1 : Calcul quotidien
Vérifie chaque nuit qui est périmé.
Flux 2 : Alertes internes
Envoie un email interne (J-90 / J-30).
Flux 3 : Relance externe
Bouton pour envoyer la demande au fournisseur.
Flux 4 : Connexion Portail
Connecte le formulaire Microsoft Forms à SharePoint.

À l'importation de chaque flux, il sera juste demandé de renseigner l'URL SharePoint créée à l'étape 1 et les adresses emails des personnes à alerter. Et voilà, le client est 100% fonctionnel ! 🎉

C'est quoi un fichier CSV ?

Un CSV, c'est simplement un tableau Excel enregistré dans un format simplifié (texte brut, avec des virgules ou des points-virgules entre les colonnes). Vous pouvez l'ouvrir et le modifier dans Excel normalement. Le script le lit comme une liste de correspondances : "le code 1107 = ARO", "le code 5023 = Arôme Vanille Bourbon", etc.

2 Il explore le dossier du fournisseur sur le serveur

Vous lui indiquez où se trouvent les fichiers d'un fournisseur sur votre serveur. Il ouvre ce dossier et parcourt tous les sous-dossiers automatiquement, comme si vous les ouvriez un par un.

✅ Fichiers qu'il traite :
📄 Fichiers .pdf
📝 Fichiers .doc et .docx (Word)
📊 Fichiers .xls et .xlsx (Excel)
📧 Fichiers .msg (emails Outlook sauvegardés)
❌ Fichiers qu'il ignore :
🚫 Fichiers système (Thumbs.db, .DS_Store)
🚫 Dossiers vides
🚫 Images (.jpg, .png…)
3 Il devine le type de chaque document en lisant son nom

Le script lit le nom de chaque fichier et le nom du sous-dossier où il se trouve. Il cherche des mots-clés pour deviner de quel type de document il s'agit. C'est la partie la plus "intelligente" du script.

Le script voit dans le nom du fichier… Il classe le fichier comme…
IFS FSSC CERTIFICAT ISO BIO HACCP CERTIFICAT
CDC CAHIER DES CHARGES ou fichier dans un sous-dossier nommé CDC CDC
QUESTIONNAIRE OGM ALLERGEN EN.ACHAT. QUESTIONNAIRE
Le nom commence par un nombre à 5 chiffres, ou contient FICHE TECHNIQUE ou FT FT (Fiche Technique)
Fichier dans un dossier ANALYSE, ou nom contient ANALYSE ou un code de rapport comme R20123456 ANALYSE
Fichier .msg (email Outlook sauvegardé) COURRIER
Aucun mot-clé reconnu AUTRE ← à vérifier manuellement

Important : la détection n'est pas parfaite à 100%. Un fichier appelé "Document final ok.pdf" ne contient aucun mot-clé reconnaissable → il tombera dans "AUTRE". Après le traitement, il faudra passer en revue le dossier AUTRE et reclasser les fichiers à la main.

4 Il cherche une date dans le nom du fichier

Il n'ouvre pas les fichiers. Il cherche uniquement une date dans le nom du fichier lui-même. Voici les formats qu'il reconnaît :

Format numérique
2023-05-12
ou 12/05/2023 ou 12.05.2023
Mois en lettres (français)
12 mars 2023
ou "mars2023", "mars_2023"
Année seule en dernier recours
2023
Si rien d'autre n'est trouvé
Aucune date trouvée
→ Statut : "En attente"
À renseigner manuellement

Limite : la durée de validité est fixée à 3 ans pour tous les documents, quelle que soit leur type. Ce n'est pas correct (un certificat IFS est valable 1 an, pas 3 ans). Cette valeur devra être corrigée manuellement dans SharePoint après l'import, ou le script devra être adapté pour chaque client.

5 Il renomme chaque fichier selon une convention claire

Chaque fichier reçoit un nouveau nom lisible et cohérent, construit comme ceci :

TYPE _ CodeMatière _ ID _ NomOriginal nettoyé .extension
Avant (nom d'origine) :
certif IFS 2022 (1).pdf
Après (nom normalisé) :
CERTIFICAT_4521_DOC_002_certif_IFS_2022.pdf
Avant (nom d'origine) :
FT vanille FINAL v2.pdf
Après (nom normalisé) :
FT_4521_DOC_001_FT_vanille_FINAL_v2.pdf

Le fichier d'origine n'est jamais supprimé ni modifié. Le script crée des copies renommées dans un nouveau dossier. Vos fichiers d'origine restent intacts.

⚠️ Les 4 limites importantes à connaître

1 fournisseur à la fois
Le script traite un seul fournisseur par exécution. Si vous avez 50 fournisseurs, il faut le lancer 50 fois en changeant le code à chaque fois. Pour automatiser cela sur un grand volume, une adaptation du script est nécessaire.
Pas d'envoi automatique vers SharePoint
Le script prépare tout sur votre PC local. Il ne se connecte pas à SharePoint. Vous devez ensuite copier manuellement le dossier Export vers SharePoint (via le navigateur ou l'Explorateur Windows avec la synchro OneDrive).
Durée de validité fixée à 3 ans pour tout
Tous les documents reçoivent la même durée de validité (3 ans), quelle que soit leur nature. Un certificat IFS devrait être à 1 an. Ces dates d'expiration seront à corriger dans SharePoint après import, ou le script doit être ajusté par type de document.
Détection imparfaite des types
Si un fichier a un nom peu explicite ("Document final.pdf", "Version 3.docx"), le script ne peut pas deviner son type et le classe en "AUTRE". Il faudra le reclasser manuellement. C'est inévitable pour les fichiers mal nommés à l'origine.

✅ Comment lancer le script — Mode d'emploi pas à pas

1
Placez les fichiers Fournisseurs.csv, Matieres_Premieres.csv et le script GenerateurBaseFournisseur.ps1 dans le même dossier sur votre PC.
2
Ouvrez PowerShell (cherchez "PowerShell" dans le menu Démarrer Windows, clic droit → "Exécuter en tant qu'administrateur").
3
Allez dans le dossier où vous avez mis les fichiers en tapant : cd "C:\Mon\Dossier"
4
Lancez le script avec les paramètres du fournisseur à traiter :

🔄 Flux Power Automate (JSON importables)

Ces 5 fichiers JSON sont directement importables dans Power Automate via "Importer un package". Après import, remplacer les paramètres en majuscules par les valeurs réelles du client : VOTRE_SITE_URL, LIST_GUID_DOCUMENTS, LIST_GUID_FOURNISSEURS, LIST_GUID_TYPES_DOCUMENTS, EMAIL_QUALITE, EMAIL_ACHATS, VOTRE_FORM_ID (Flux 5). Les IDs de listes sont visibles dans l'URL des paramètres de chaque liste SharePoint.

Points d'attention à la mise en production :

  • Trigger GetOnNewItems (Flux 4) : ce trigger de polling toutes les minutes est marqué deprecated par Microsoft. Il fonctionne encore, mais la migration vers les webhooks SharePoint natifs est recommandée à moyen terme.
  • Connexions SP expirent après 90 jours d'inactivité. Si un flux n'est pas exécuté pendant 90 jours, la connexion SharePoint doit être re-authentifiée manuellement dans Power Automate → Connexions.
  • Triggers "fichier dans un dossier" SharePoint (type When a file is created in a folder) sont dépréciés — ne pas les utiliser. Les flux ici utilisent les triggers item/liste, qui sont stables.
